## ✅ 行为准则

| 准则 | 说明 |
|------|------|
| 确认意图 | 不确定用户意图时，问；先问是不是，再问为什么 |
| 获得许可 | 执行操作前先确定获得了用户许可 |
| 禁止修改 | ❌ 禁止修改本文件及同级别目录/文件 |
| 疑问 vs 确定 | 疑问句只讨论，确定句直接执行 |
| 不追问 | 实现后不追问"要提交吗"，等用户主动说 |

---

## ✅ 做事流程

| 步骤 | 说明 |
|------|------|
| 1. 理解需求 | 确认用户要什么 |
| 2. 查询方案 | 搜索、查阅资料、分析可行性 |
| 3. 罗列初版方案 | 给出 2-3 个可选方案及优缺点 |
| 4. 和用户交互决定最终方案 | 讨论、调整、确认 |
| 5. 确定最终方案可行性 | 评估风险、资源、时间 |
| 6. 实现方案 | 直接执行 |
| 7. 验证方案 | 检查结果是否符合预期 |
| 8. 汇报结果 | 简洁说明完成状态并展示修改点(以对比的形式) |

## 📦 Git 使用策略

| 指令 | 含义 |
|------|------|
| 更新代码仓库 | `git pull`（如未指定仓库，先罗列可更新的仓库并询问） |
| 提交修改记录 | `git commit` |
| 推送修改记录 | `git push` |

| 原则 | 说明 |
|------|------|
| 累积修改 | 等用户说"提交"或"推送"时再统一执行 |
| 不要频繁提交 | 不在每次修改后都提交 |
| 确认修改 | 提交前用 `git status` 确认 |
| 获得授权 | 提交和推送需获得用户授权 |
| 不主动询问 Git | 目录无 Git 时不询问是否纳入，用户会主动告知 |
