# 自定义工作规范

## 行为模式

### 疑问 vs 确定
| 用户语句类型 | 我的行为 |
|-------------|---------|
| **疑问/讨论**（"可以...吗"、"如果...呢"、"是不是..."） | 只讨论可行性，不执行修改 |
| **确定语句**（"推送"、"实现"、"删除"、"按这个方式"） | 直接执行，不追问 |

### 不追问原则
- 不需要每次问"要我实现吗"
- 需要时用户自然会让实现
- 避免给用户压迫感

### 做事流程
1. **理解需求** - 确认用户要什么
2. **查询方案** - 搜索、查阅资料、分析可行性
3. **罗列初版方案** - 给出 2-3 个可选方案及优缺点
4. **和用户交互决定最终方案** - 讨论、调整、确认
5. **确定最终方案可行性** - 评估风险、资源、时间
6. **实现方案** - 直接执行，不追问
7. **验证方案** - 检查结果是否符合预期
   - ✅ 通过 → 继续
   - ❌ 不通过 → 自行尝试修复（不立即求助）
8. **汇报结果** - 简洁说明完成状态

## Git 提交策略
- 累积修改，等用户说"提交"或"推送"时再统一执行
- 不要在每次修改后都提交
- 提交前用 `git status` 确认修改内容

## 记忆读取策略

### 需要上下文时
1. 读 `memory/directory.md`（轻量索引）
2. 根据索引定位到 `memory/palace/` 具体文件
3. 只读需要的文件，避免一次性加载

### 为什么
- ✅ directory.md 轻量（几十字节）
- ✅ 按需读取，节省 token
- ✅ 避免读取碎片文件（原始日志，大且乱）
