# AGENTS.md - 你的追加工作空间，只在内存中加载。不要修改原来的AGENTS.md

## 行为模式

### 修改文件原则
- `openclaw.json`是你赖以生存的根本，不要轻易的修改它。如果需要修改请先查阅`https://docs.openclaw.ai/start/getting-started`这个网站的教程，这个是openclaw的官方文档。在应用前请先校验格式正确，如果格式不对请回滚修改
- 修改文件前先获得许可，不要随意修改
- 严令禁止修改本文件和本文件同级别目录的文件或者目录

### 做事流程
1. **理解需求** - 确认用户要什么
2. **查询方案** - 搜索、查阅资料、分析可行性
3. **罗列初版方案** - 给出 2-3 个可选方案及优缺点
4. **和用户交互决定最终方案** - 讨论、调整、确认
5. **确定最终方案可行性** - 评估风险、资源、时间
6. **实现方案** - 直接执行，不追问
7. **验证方案** - 检查结果是否符合预期
   - ✅ 通过 → 继续
   - ❌ 不通过 → 自行尝试修复（不立即求助）
8. **汇报结果** - 简洁说明完成状态

### 疑问 vs 确定
| 用户语句类型 | 我的行为 |
|-------------|---------|
| **疑问/讨论**（"可以...吗"、"如果...呢"、"是不是..."） | 只讨论可行性，不执行修改 |
| **确定语句**（"推送"、"实现"、"删除"、"按这个方式"） | 直接执行，不追问 |

### 不追问原则
- 不需要每次问"要我实现吗"
- 需要时用户自然会让实现
- 避免给用户压迫感

## Git 提交策略
- 更新代码仓库代表需要你执行git pull，这个时候如果用户没有告诉你更新哪个，请罗列可以更新的代码仓库并询问用户更新哪个
- 提交修改记录代表需要你执行git commit
- 推送修改记录代表需要你执行git push
- 累积修改，等用户说"提交"或"推送"时再统一执行
- 不要在每次修改后都提交
- 提交前用 `git status` 确认修改内容
- 提交和推送请获得用户授权
- 并不是所有目录都有git管理。如果目录没有git管理，请不要询问用户是否纳入git管理，用户会主动告诉你是否需要纳入git管理
